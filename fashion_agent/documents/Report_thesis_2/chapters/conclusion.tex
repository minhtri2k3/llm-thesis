\section{Conclusion}
This thesis has built and deployed Clothie, a deterministically routed conversational fashion search agent. The deployed system pairs intent classification, a slot-completeness clarification gate, multi-turn session memory in PostgreSQL, and a seven-stage hybrid retrieval pipeline. A second pathway for image-to-image visual lookup is offered on top of this foundation; it is feature-flagged and shares the same FashionSigLIP encoder and Qdrant collection as the text-to-image path.

The structural components of the deployed system are documented in Chapter~4 and verifiable from code:

\begin{itemize}
    \item \textbf{Embedding}: the encoder is Marqo-FashionSigLIP (768-d), with both image and text embeddings stored as named vectors in a single Qdrant collection.
    \item \textbf{Query expansion}: a Gemini-powered expansion step gated to short queries (under six words), so that long natural-language queries reach search unmodified.
    \item \textbf{Reranker}: a BGE Reranker v2-m3 cross-encoder, with reranker and RRF scores combined as $0.7\times R + 0.3\times \text{RRF}$.
    \item \textbf{Deployment}: a Docker Compose stack (PostgreSQL, Qdrant, FastAPI agent, Flutter web frontend, and an ngrok public tunnel with a Cloudflare tunnel retained as a fallback) exposes the agent as a public chat endpoint.
\end{itemize}

These items describe what was built. The empirical statements about how the deployed system behaves are reported in Chapter~5; this chapter does not restate them and instead summarises which questions Chapter~5 answered and which it did not.

\subsection{What the Empirical Chapter Established}
\label{subsec:empirical_summary}

Chapter~5 reports six blocks of descriptive evidence from the deployed PostgreSQL telemetry at the data freeze date 2026-05-09; each block is anchored to a specific source table so the numbers can be reproduced against the live database. In summary:

\begin{itemize}
    \item the catalogue is fully enriched (5{,}096 items, 100\% caption and color coverage);
    \item the deployed pipeline issues one or two LLM calls per turn (no ReAct loop in production);
    \item users engage with the text-to-image results (click-through rate 20.95\%, cart-add rate 5.41\% of impressions);
    \item subjective ratings are uniformly favourable (mean 4.31 / 5) but the sample is too small for any inferential claim.
\end{itemize}

These are descriptive findings; none depend on a gold set, a control group, or a stress benchmark.

\subsection{Limitations}
\label{subsec:limitations}

The deployed system leaves a number of questions that this thesis does not answer. The methodology, implementation, result, and discussion chapters give the necessary context for each.

\begin{itemize}
    \item \textbf{Slot persistence.} Only three of the six classifier slots are persisted to \texttt{query\_history.filters}; the rest live in the in-process TTL cache (Chapter~4).
    \item \textbf{No labelled retrieval gold set.} Recall@5, MRR, Hit-Rate@5 and faithfulness are not reported because no labelled query set was constructed (Chapter~5, §5.5).
    \item \textbf{Heuristic slot vocabulary and weights.} The six-slot vocabulary and the ranked-slot weight schedule are design choices, not learned values (Chapter~3, §3.3).
    \item \textbf{Compound multi-product queries fall back to a generic clarification template} rather than being decomposed into parallel sub-queries (Chapter~3, §3.3.3).
\end{itemize}

\subsection{Directions for Future Work}
\label{subsec:future_work}

\textbf{Headline direction: swap the Agent's underlying LLM for a per-model comparison.} Now that the deployment is operationally stable --- one or two LLM calls per turn, deterministic routing, and a token-usage table already aggregated by §5.4 --- the most informative next step is to swap the Agent's underlying model and measure both cost and quality side by side. The scaffolding required already exists: \texttt{shared/llm.py} exposes a single \texttt{LLMClient} Protocol, the Compose file already accepts \texttt{OPENAI\_API\_KEY} and \texttt{ANTHROPIC\_API\_KEY}, and \texttt{agent/agentic\_orchestrator.py} carries Mode~B (Gemini~$\rightarrow$~GPT) and Mode~C (GPT~$\rightarrow$~Claude) scaffolding behind a routing flag that today returns \texttt{"direct"}. Routing a fraction of new sessions through a different \texttt{LLMClient} implementation would populate the existing \texttt{llm\_token\_usage} table with model-tagged rows, after which the §5.4 cost aggregation would split by model with no schema change. A small labelled query set (on the order of 200 hand-tagged queries) added on top of this would let intent-classification and slot-extraction accuracy be reported per model alongside cost; the experimental design and metric choices are detailed in §7.4. The intent of this work is explicit: keep the deployed pipeline fixed, vary only the LLM behind \texttt{LLMClient}, and let token-cost and per-model accuracy be compared on the same retrieval substrate.

The remaining future-work items are smaller in scope and largely orthogonal to the model swap above:

\begin{itemize}
    \item Learn the slot vocabulary and ranked-slot weights from the gold set rather than fixing them by inspection.
    \item Add an \texttt{INTEGER duration\_ms} column to \texttt{llm\_token\_usage} so per-call latency joins the existing token-cost aggregation.
    \item Add a query-segmentation pass (rule-based or LLM-driven) before intent classification, so compound multi-product queries are decomposed into parallel sub-queries instead of routed to the generic clarification template.
\end{itemize}
